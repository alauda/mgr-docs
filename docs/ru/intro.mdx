Alauda Database для MySQL-MGR - это решение для высокодоступной базы данных, основанное на технологии групповой репликации MySQL и работающие на Kubernetes. Оно предоставляет возможности автоматизированного развертывания и управления кластерами группы репликации MySQL.

## Ключевые особенности

- **Высокая доступность**: Архитектура кластера на основе групповой репликации MySQL, обеспечивающая автоматическую синхронизацию данных между несколькими узлами и самовосстановление при сбоях, с автоматическим переключением при сбое основного узла, что снижает риски прерывания сервиса.
- **Согласованность данных**: Распределенная аутентификация на основе технологии групповой репликации, обеспечивающая согласованность данных.
- **Автоматизированное управление**: Реализация управления полным жизненным циклом развертывания кластера, обновления конфигурации, восстановления резервных копий и т.д. через Operator, систематически снижающее операционные расходы.
- **Эластичное масштабирование**: Поддержка онлайн-добавления или удаления узлов, автоматическая синхронизация состояния глобальных транзакций, адаптация к динамическим изменениям нагрузки.
- **Разделение чтения и записи**: Поддержка разделения чтения и записи через MySQL Router.

## Сценарии применения

- Критически важные бизнес-базы данных, требующие высокой доступности.
- Распределенные базы данных, требующие сильной согласованности данных.
- Облачные базы данных, нуждающиеся в эластичном масштабировании.

## Ограничения по использованию

- Количество участников группы: Максимальное количество участников группы репликации MySQL составляет 9. Рекомендуется как минимум 3 участника для обеспечения высокой доступности.
- Движок InnoDB: Зависит от движка хранения InnoDB; рекомендуется сконфигурировать `disabled_storage_engines="MyISAM,BLACKHOLE,FEDERATED,ARCHIVE,MEMORY"` для отключения неподдерживаемых движков хранения.
- Явный первичный ключ: Каждая таблица для репликации должна иметь определенный первичный ключ или эквивалент первичного ключа, где эквиваленты - это уникальные ключи, не содержащие нулевых значений. Настройте `sql_generate_invisible_primary_key: ON`, чтобы поддерживать таблицы без явно определенных первичных ключей.
- Качество сети: Спроектирована для развертывания в кластерных средах, где серверные экземпляры расположены очень близко друг к другу, что обеспечивает низкую задержку сети. Высокая задержка сети может привести к задержкам в репликации или даже к сбоям в репликации.
- Ограничения уровня изоляции: Рекомендуется использовать уровень изоляции READ COMMITTED; уровень изоляции SERIALIZABLE не поддерживается.
- Ограничения внешних ключей: Рекомендуется избегать каскадных внешних ключей в режиме многомастера.
- Взаимные блокировки в многомастере: В режиме многомастера могут возникнуть взаимные блокировки при использовании операторов `SELECT .. FOR UPDATE`, поскольку блокировки не разделяются между участниками.
- Конфликты параллельных DDL и DML: В режиме многомастера операции DDL и DML могут вызывать конфликты; рекомендуется избегать параллельных операций DDL и DML.
- Ограничения размера транзакций: Убедитесь, что размер транзакции находится в пределах `group_replication_transaction_size_limit` (по умолчанию примерно 143 МБ) и может быть завершен в течение периода `group_replication_member_expel_timeout` (по умолчанию 5 с).
- Ограничения по гетерогенной архитектуре: В силу ограничений плагина MySQL Clone в настоящее время не поддерживаются гетерогенные сценарии.
- Ограничения по последовательному обновлению: MySQL Pods в первую очередь будут перезапускать вторичных участников, первичный участник будет перезапущен последним. Этот процесс может вызвать короткие прерывания подключения, требуя механизмов повторной попытки на уровне приложения.
