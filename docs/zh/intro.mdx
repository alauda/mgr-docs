# 介绍

Alauda Database for MySQL-MGR 是基于 MySQL Group Replication 技术的高可用数据库解决方案，运行在 Kubernetes 上。它提供了自动化部署与管理 MySQL Group Replication 集群的能力。

## 核心优势

- **高可用性**：基于 MySQL Group Replication 提供多主复制和高可用性
- **自动化管理**：通过 Operator 自动化管理 MySQL 集群
- **弹性扩展**：支持在线扩展和缩容
- **数据一致性**：基于 Paxos 协议保证数据强一致性
- **读写分离**：支持通过 MySQL Router 实现读写分离

## 应用场景

- 需要高可用性的关键业务数据库
- 需要多主写入的分布式数据库场景
- 需要弹性扩展的云原生数据库

## 使用限制

- 组成员数量: 复制组成员的 MySQL 最大数量为9, 建议至少3个成员，以保证高可用性。
- innodb引擎: 依赖 Innodb 存储引擎, 建议配置 `disabled_storage_engines="MyISAM,BLACKHOLE,FEDERATED,ARCHIVE,MEMORY"` 禁用不支持的存储引擎。
- 显式主键: 组要复制的每个表都必须具有定义的主键或等效的主键, 其中等效项是非 null 唯一键, 可通过配置 `sql_generate_invisible_primary_key: ON` 以支持无显式指定主键的数据表。
- 网络质量: 旨在部署于服务器实例非常接近的集群环境, 彼此之间的网络延迟很低。如果网络延迟很高，可能会导致复制延迟，甚至复制失败。
- 隔离级别限制: 建议使用 READ COMMITTED 隔离级别，不支持 SERIALIZABLE 隔离级别。
- 外键限制: 多主模式不建议使用级联外键。
- 多主模式死锁: 多主模式下，由于锁不在成员中共享, 使用 `SELECT .. FOR UPDATE` 语句可能导致死锁。
- 并发 DDL 与 DML 冲突: 多主模式下，DDL 与 DML 操作可能导致冲突，建议避免并发 DDL 与 DML 操作。
- 事务大小限制: 确保事务大小在 group_replication_transaction_size_limit 范围内(默认大约 143 MB), 并可以在  group_replication_member_expel_timeout 时间内(默认5s)完成。
- 异构架构限制: 由于 MySQL Clone 插件限制，暂不支持异构场景。
